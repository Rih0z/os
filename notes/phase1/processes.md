# Phase 1: プロセス管理

## 初心者向け：基本概念の例え話

### コンテキストスイッチって何？

```
┌─────────────────────────────────────────────────────────────┐
│  本を読んでいる時の「しおり」の役割                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  あなたは3冊の本を同時に読んでいます：                      │
│                                                             │
│  本A（小説）     本B（教科書）    本C（漫画）              │
│  ┌─────┐       ┌─────┐        ┌─────┐                    │
│  │ 42頁│       │ 15頁│        │ 88頁│                    │
│  └─────┘       └─────┘        └─────┘                    │
│                                                             │
│  本を切り替える時、何が必要？                               │
│  1. 「どこまで読んだか」を覚える（しおりを挟む）           │
│  2. 次の本を開く                                           │
│  3. しおりのページから読み始める                           │
│                                                             │
│  これが「コンテキストスイッチ」です！                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### プロセス構造体って何？

```
┌─────────────────────────────────────────────────────────────┐
│  プロセス構造体 = 「プロセスの個人情報カード」              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、学校の生徒一人一人に「個人情報カード」があります   │
│                                                             │
│  ┌─────────────────────────────────────┐                   │
│  │  氏名：田中 太郎                    │                   │

│  │  クラス：2年A組                     │                   │
│  └─────────────────────────────────────┘                   │
│                                                             │
│  同じように、プロセス一人一人に「情報カード」があります     │
│                                                             │
│  ┌─────────────────────────────────────┐                   │
│  │  プロセス名：init                   │                   │
│  │  プロセス番号：1                    │                   │
│  │  優先度：7                          │                   │
│  │  状態：実行中                       │                   │
│  └─────────────────────────────────────┘                   │
│                                                             │
│  これが「プロセス構造体」です！                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### スケジューラって何？

```
┌─────────────────────────────────────────────────────────────┐
│  スケジューラ = 「次に誰を実行するか決める人」              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、先生が生徒を指名する時：                           │
│                                                             │
│  先生「次は...田中さん！」                                  │
│                                                             │
│  どうやって決める？                                         │
│  - 手を挙げた人から選ぶ？（ラウンドロビン）                │
│  - 成績が良い人から選ぶ？（優先度ベース）                  │
│                                                             │
│  OSでも同じ！                                               │
│  「次はどのプロセスを実行する？」を決めるのがスケジューラ   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 用語の読み方と意味（超初心者向け）

| 用語 | 読み方 | 例え | 意味 |
|------|--------|------|------|
| **レジスタ** | れじすた | 計算機 | CPU内の「超高速メモリ」 |
| **スタック** | すたっく | 付箋の山 | 一時的なデータを積み上げる場所 |
| **プログラムカウンタ** | - | しおり | 「次の命令」の場所を覚える |
| **コンテキスト** | - | 読書の状態 | 「どこまで実行したか」の全情報 |
| **スイッチ** | - | 切り替え | 入れ替えること |
| **セグメント** | - | 区画 | メモリを区切った領域 |
| **割り込み** | - | 割り込み電話 | 「今すぐ対応して！」という合図 |

## IPC（Inter-Process Communication）って何？

### IPCの略語の意味

```
┌─────────────────────────────────────────────────────────────┐
│  IPC = Inter-Process Communication                          │
│        ↓          ↓            ↓                            │
│      プロセス間   通信                                      │
│                                                             │
│  読み方：アイピーシー                                       │
│  意味：プロセスとプロセスが情報をやり取りする仕組み         │
└─────────────────────────────────────────────────────────────┘
```

### 初心者向け：IPCの例え話

```
┌─────────────────────────────────────────────────────────────┐
│  IPC = プロセス間での「手紙のやり取り」                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、別々の部屋にいる2人の人が会話したい時：            │
│                                                             │
│  部屋A（田中さん）          部屋B（佐藤さん）               │
│  ┌───────┐                  ┌───────┐                      │
│  │       │                  │       │                      │
│  │ 田中  │   手紙を投函 →   │ 佐藤  │                      │
│  │       │   ← 手紙を返信   │       │                      │
│  └───────┘                  └───────┘                      │
│                                                             │
│  直接話せないから「手紙」でやり取りする                     │
│                                                             │
│  OSでも同じ！                                               │
│  - プロセスは「別々の部屋」にいる                           │
│  - メモリが独立しているから直接話せない                     │
│  - IPC（メッセージ）でやり取りする                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### なぜIPCが必要？

```
┌─────────────────────────────────────────────────────────────┐
│  プロセスは「独立した世界」に住んでいる                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  プロセスAの世界：                                          │
│  ┌─────────────────────────────────────┐                   │
│  │ 自分のメモリしか見えない             │                   │
│  │ 自分の変数しか使えない               │                   │
│  │ 他のプロセスのことは知らない         │                   │
│  └─────────────────────────────────────┘                   │
│                                                             │
│  でも、連携したい時がある：                                 │
│  - ファイルを読みたい → ファイルシステムに頼む             │
│  - 時刻を知りたい → 時計プロセスに頼む                     │
│  - 画面に表示したい → 画面プロセスに頼む                   │
│                                                             │
│  そこでIPCを使う！                                          │
│  「お願い、これやって！」とメッセージを送る                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### MINIX 3のIPCの種類

```
┌─────────────────────────────────────────────────────────────┐
│  MINIX 3のIPC = 3つの基本操作                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. send() = 「手紙を送る」                                 │
│     - 相手が受け取るまで待つ                                │
│     - 例：「ファイルを開いて」と依頼する                   │
│                                                             │
│  2. receive() = 「手紙を受け取る」                          │
│     - 誰かからメッセージが来るまで待つ                      │
│     - 例：「依頼が来るのを待つ」                            │
│                                                             │
│  3. sendrec() = 「手紙を送って、返事を待つ」                │
│     - send() + receive() の組み合わせ                       │
│     - 例：「ファイルを開いて」→「開きました」               │
│                                                             │
│  4. notify() = 「通知だけ送る」                             │
│     - 相手を待たない                                        │
│     - 例：「イベントが起きたよ」とだけ知らせる             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### MINIX 3のIPCの仕組み

```
┌─────────────────────────────────────────────────────────────┐
│  メッセージの流れ                                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  プロセスA（クライアント）    プロセスB（サーバー）         │
│  ┌─────────────┐              ┌─────────────┐              │
│  │             │              │             │              │
│  │ 1. send()   │ ──メッセージ→│ 2. receive()│              │
│  │    「依頼」 │              │    「受取」 │              │
│  │             │              │             │              │
│  │ 4. receive()│←─メッセージ──│ 3. send()   │              │
│  │    「待機」 │              │    「返信」 │              │
│  └─────────────┘              └─────────────┘              │
│                                                             │
│  重要：                                                     │
│  - send() は相手が受け取るまでブロック（待機）する          │
│  - receive() はメッセージが来るまでブロックする             │
│  - これを「同期メッセージパッシング」と呼ぶ                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### IPCの状態遷移

```
┌─────────────────────────────────────────────────────────────┐
│  プロセスの状態とIPC                                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  実行可能（RUNNABLE）                                       │
│  │                                                         │
│  │ send() して相手が待っていない                           │
│  ↓                                                         │
│  送信待ち（SENDING）                                        │
│  │                                                         │
│  │ 相手が receive() した                                   │
│  ↓                                                         │
│  実行可能（RUNNABLE）                                       │
│                                                             │
│  ─────────────────────────────────────────────             │
│                                                             │
│  実行可能（RUNNABLE）                                       │
│  │                                                         │
│  │ receive() してメッセージがない                          │
│  ↓                                                         │
│  受信待ち（RECEIVING）                                      │
│  │                                                         │
│  │ 誰かが send() した                                      │
│  ↓                                                         │
│  実行可能（RUNNABLE）                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## MINIX 3のプロセス構造体

### 構造体の主要なフィールド

```c
struct proc {
  struct stackframe_s p_reg;    // 保存されたレジスタ（しおり）
  proc_nr_t p_nr;               // プロセス番号（学籍番号）
  char p_rts_flags;             // 実行状態フラグ（出席/欠席）
  char p_priority;              // 現在の優先度（成績）
  char p_max_priority;          // 最大優先度（成績の上限）
  char p_ticks_left;            // 残り時間量子（残り時間）
  char p_quantum_size;          // 時間量子サイズ（持ち時間）
  struct proc *p_nextready;     // 次の準備完了プロセス（次の人）
  message *p_messbuf;           // IPCメッセージバッファ（メール）
  char p_name[P_NAME_LEN];      // プロセス名（氏名）
};
```

### MikanOSとの比較

| 項目 | MikanOS | MINIX 3 |
|------|---------|---------|
| タスク構造体 | `struct TaskContext` | `struct proc` |
| スケジューリング | ラウンドロビン | 優先度ベース |
| 優先度 | なし | 16段階（0=最高） |
| 時間量子 | なし | あり（`p_ticks_left`） |
| IPC | メッセージパッシング | `send()`/`receive()` |

## 8ビットフラグ管理

### 8ビットとは？

```
8ビット = 8つの0と1の並び

例: 0 1 0 0 0 1 0 0
    ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑
    ビット7〜ビット0

8ビットで表せる数：0 〜 255（全部で256通り）
```

### なぜ8ビットで状態を管理するのか？

**1つの整数で「複数の状態」を同時に表せるから！**

```
┌─────────────────────────────────────────────────────┐
│  8ビットの各ビットに「意味」を割り当てる              │
├─────────────────────────────────────────────────────┤
│                                                     │
│    ビット番号:  7   6   5   4   3   2   1   0       │
│                ┌───┬───┬───┬───┬───┬───┬───┬───┐   │
│    値:         │ 0 │ 0 │ 0 │ 0 │ 0 │ 0 │ 0 │ 0 │   │
│                └───┴───┴───┴───┴───┴───┴───┴───┘   │
│                              ↑       ↑   ↑         │
│                              │       │   │         │
│                    受信待ち ─┘       │   └─ スロット空き │
│                              送信待ち ─┘              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### MINIX 3のフラグ定義

```c
// MINIX 3 kernel/proc.h
#define SLOT_FREE   0x01  // ビット0: スロット空き
#define NO_MAP      0x02  // ビット1: メモリマップなし
#define SENDING     0x04  // ビット2: 送信待ち
#define RECEIVING   0x08  // ビット3: 受信待ち
```

### 16進数とビットの対応表

```
┌─────────┬───────────┬────────────────────────────────┐
│ 16進数  │  2進数    │ ビット位置                      │
├─────────┼───────────┼────────────────────────────────┤
│  0x01   │ 0000 0001 │ ビット0 ← SLOT_FREE            │
│  0x02   │ 0000 0010 │ ビット1 ← NO_MAP               │
│  0x04   │ 0000 0100 │ ビット2 ← SENDING              │
│  0x08   │ 0000 1000 │ ビット3 ← RECEIVING            │
│  0x10   │ 0001 0000 │ ビット4 ← SIGNALED             │
│  0x20   │ 0010 0000 │ ビット5 ← SIG_PENDING          │
│  0x40   │ 0100 0000 │ ビット6 ← P_STOP               │
│  0x80   │ 1000 0000 │ ビット7 ← NO_PRIV              │
└─────────┴───────────┴────────────────────────────────┘

法則: 0x01, 0x02, 0x04, 0x08, 0x10, 0x20, 0x40, 0x80
      = 2⁰, 2¹, 2², 2³, 2⁴, 2⁵, 2⁶, 2⁷
```

## 時間量子（Quantum）とティック（Tick）

### 時間量子って何？

```
┌─────────────────────────────────────────────────────────────┐
│  時間量子 = 「1回の持ち時間」                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、ゲームの持ち時間：                                 │
│                                                             │
│  プレイヤーA: ████████ (8秒)                               │
│              ↓ 時間切れ！                                   │
│  プレイヤーB: ████████ (8秒)                               │
│              ↓ 時間切れ！                                   │
│  プレイヤーC: ████████ (8秒)                               │
│                                                             │
│  OSでも同じ：                                               │
│  - プロセスは「時間量子」だけCPUを使える                    │
│  - 時間切れしたら次のプロセスへ                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### ティックって何？

```
┌─────────────────────────────────────────────────────────────┐
│  ティック = 「OSの時間の最小単位」                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、時計の秒針：                                       │
│                                                             │
│  1秒 → 2秒 → 3秒 → 4秒 → 5秒 → 6秒 → 7秒 → 8秒            │
│  ↓    ↓    ↓    ↓    ↓    ↓    ↓    ↓                      │
│  tick tick tick tick tick tick tick tick                    │
│  0    1    2    3    4    5    6    7                       │
│                                                             │
│  ↑                                                         │
│  8ティック = 1時間量子                                      │
│                                                             │
│  MINIX 3の場合：                                            │
│  - 1ティック = 10ミリ秒                                     │
│  - 8ティック = 80ミリ秒 = 1時間量子                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 各OSでのティック設定

| OS | 1ティック | 設定場所 |
|---|---|---|
| **MINIX 3** | 約10ms | `HZ = 100`（1秒に100回） |
| **Linux** | 1-10ms | `CONFIG_HZ`（100, 250, 1000） |
| **Windows** | 約15.6ms | デフォルト |
| **MikanOS** | なし | タイマー割り込みなし |

## ラウンドロビン vs 優先度ベース

### ラウンドロビン（MikanOS）

```
┌─────────────────────────────────────────────────────────────┐
│  ラウンドロビン = 「順番に実行するだけ」                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、順番に発言する会議：                               │
│                                                             │
│  Aさん → Bさん → Cさん → Aさん → ...                       │
│                                                             │
│  メリット:                                                  │
│  - 簡単！                                                   │
│  - 公平！                                                   │
│                                                             │
│  デメリット:                                                │
│  - 重要な人が待たされる                                     │
│  - 緊急の発言ができない                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 優先度ベース（MINIX 3）

```
┌─────────────────────────────────────────────────────────────┐
│  優先度ベース = 「重要な人を先に」                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、社長→部長→社員の順で発言：                        │
│                                                             │
│  社長（優先度0）: 最優先！                                  │
│  部長（優先度7）: 普通                                      │
│  社員（優先度15）: 誰もいない時だけ                         │
│                                                             │
│  メリット:                                                  │
│  - 重要な処理がすぐ実行される                               │
│  - 緊急の処理ができる                                       │
│                                                             │
│  デメリット:                                                │
│  - 複雑！                                                   │
│  - 低優先度がずっと実行されない可能性                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## MINIX 3のスケジューラ

### enqueue() - 「列に並ぶ」

```
┌─────────────────────────────────────────────────────────────┐
│  enqueue = 「列に並ぶ」                                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、お店のレジに並ぶ時：                               │
│                                                             │
│  列の先頭 ← [客A] [客B] [客C] → 列の末尾                   │
│                                                             │
│  新しい客Dが来たら？                                        │
│  - 列の末尾に並ぶ                                           │
│                                                             │
│  MINIX 3でも同じ：                                          │
│  - 実行可能になったプロセスを「列」に追加                   │
│  - 優先度ごとに別々の列がある                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### pick_proc() - 「次の人を選ぶ」

```
┌─────────────────────────────────────────────────────────────┐
│  pick_proc = 「次の人を選ぶ」                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、先生が指名する時：                                 │
│                                                             │
│  先生「一番優秀な人から指名します」                         │
│  → 成績が一番良い人を選ぶ                                   │
│                                                             │
│  MINIX 3でも同じ：                                          │
│  → 優先度0（最高）から順にチェック                         │
│  → 最初に見つかった人を選ぶ                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### MINIX 3のスケジューラ図解

```
┌─────────────────────────────────────────────────────────────┐
│  優先度キュー（rdy_head[]）                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  q=0 (TASK_Q)    ┌───┐                                     │
│                  │ A │ → 実行中                            │
│                  └───┘                                     │
│                                                             │
│  q=7 (USER_Q)    ┌───┐    ┌───┐    ┌───┐                  │
│                  │ B │ → │ C │ → │ D │ → NIL              │
│                  └───┘    └───┘    └───┘                  │
│                                                             │
│  q=15 (IDLE_Q)   ┌───┐                                     │
│                  │ E │ → NIL                               │
│                  └───┘                                     │
│                                                             │
│  pick_proc() の動き:                                        │
│  1. q=0 をチェック → Aが見つかる → Aを選択                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Rustでの実装

### プロセス状態フラグ（ProcessFlags）

```rust
pub struct ProcessFlags(u8);

impl ProcessFlags {
    /// ビット0: スロットは空き
    pub const SLOT_FREE: u8 = 0x01;
    /// ビット2: 送信待ち
    pub const SENDING: u8 = 0x04;
    /// ビット3: 受信待ち
    pub const RECEIVING: u8 = 0x08;
}
```

### 優先度（Priority）

```rust
pub struct Priority(u8);

impl Priority {
    pub const TASK_Q: u8 = 0;   // カーネルタスク（最高）
    pub const USER_Q: u8 = 7;   // ユーザープロセス
```

### スケジューラ（Scheduler）

```rust
pub struct Scheduler {
    /// 各優先度のキューの先頭
    rdy_head: [Option<usize>; NR_SCHED_QUEUES],
}

impl Scheduler {
    /// プロセスをエンキュー
    pub fn enqueue(&mut self, process_index: usize, priority: u8) {
        let q = priority as usize;
        self.rdy_head[q] = Some(process_index);
    }
    
    /// 次のプロセスを選択
    pub fn pick_next(&self) -> Option<usize> {
        // 優先度0（最高）から順にチェック
        for q in 0..NR_SCHED_QUEUES {
            if let Some(idx) = self.rdy_head[q] {
                return Some(idx);
            }
        }
        None
    }
}
```
