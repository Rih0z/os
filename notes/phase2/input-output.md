# Phase 2: Input/Output

## 初心者向け：I/Oって何？

### I/Oの例え話

```
┌─────────────────────────────────────────────────────────────┐
│  I/O = Input/Output = 入力と出力                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例えば、レストランでのやり取り：                           │
│                                                             │
│  お客さん（CPU）    ウェイター（ドライバ）    厨房（デバイス）│
│  ┌─────┐           ┌─────┐              ┌─────┐           │
│  │     │  注文する  │     │   伝票を渡す  │     │           │
│  │ 客  │ ────────► │ウェイター│ ────────► │ 厨房 │           │
│  │     │           │     │              │     │           │
│  │     │ ◄──────── │     │ ◄─────────── │     │           │
│  │     │  料理を受け取る  │   料理を受け取る│     │           │
│  └─────┘           └─────┘              └─────┘           │
│                                                             │
│  OSでも同じ！                                               │
│  - CPUは直接デバイスを操作しない                            │
│  - ドライバが仲介役                                         │
│  - ドライバがデバイスとやり取りする                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### I/Oデバイスの種類

| 種類 | 特徴 | 例 |
|------|------|-----|
| **ブロックデバイス** | ブロック単位、ランダムアクセス | HDD, SSD, USB |
| **キャラクタデバイス** | 1バイトずつ、ストリーム | キーボード, マウス |
| **ネットワークデバイス** | パケット単位 | NIC |

### 用語の読み方と意味

| 用語 | 読み方 | 意味 |
|------|--------|------|
| **I/O** | アイオー | Input/Output（入出力） |
| **デバイス** | でばいす | ハードウェア機器 |
| **ドライバ** | どらいば | デバイスを制御するソフトウェア |
| **割り込み** | わりこみ | デバイスからの通知 |
| **ポーリング** | ぽーりんぐ | 定期的にチェック |
| **DMA** | ディーエムエー | Direct Memory Access（自動転送） |

## MINIX 3のドライバアーキテクチャ

### マイクロカーネルの特徴

```
┌─────────────────────────────────────────────────────────────┐
│  MINIX 3 = ドライバもユーザー空間で動く                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  モノリシックカーネル（Linux等）                            │
│  → ドライバがクラッシュ = カーネル全体がクラッシュ          │

└─────────────────────────────────────────────────────────────┘
```

| ディレクトリ | 内容 |

## デッドロック

### デッドロックとは？

```
┌─────────────────────────────────────────────────────────────┐
│  デッドロック = お互いに待ち合って進まない状態              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  プロセスA: リソースXを持ってYを待つ                        │
│  プロセスB: リソースYを持ってXを待つ                        │
│  → 誰も動けない！                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### デッドロックの4つの条件

1. **相互排除** - リソースは同時に1つだけ
2. **保持と待機** - 保持しながら待つ
3. **剥奪不可** - 強制的に取り上げられない
4. **循環待機** - 待ち関係がサイクル

### デッドロック対策

| アプローチ | 説明 |
|-----------|------|
| 予防 | 4つの条件のどれかを崩す |
| 回避 | デッドロックが起きない要求だけ許可 |
| 検出と回復 | 検出したら解決する |
| 無視 | 稀だから無視（Linux, Windows） |

## シリアルポート（UART）ドライバの実装

### UARTとは？

```
┌─────────────────────────────────────────────────────────────┐
│  UART = Universal Asynchronous Receiver/Transmitter         │
│  （汎用非同期送受信機）                                      │
│                                                             │
│  1文字ずつ送受信する「電話回線」のようなもの                 │
│                                                             │
│  CPU ──[1バイト書き込む]──► UART ──[シリアル信号]──► 相手    │
│  CPU ◄──[1バイト読み出す]── UART ◄──[シリアル信号]── 相手    │
│                                                             │
│  QEMUでは、UARTに書いた文字がターミナルに表示される          │
└─────────────────────────────────────────────────────────────┘
```

### UART 16550A のレジスタ

| オフセット | レジスタ名 | 用途 |
|-----------|-----------|------|
| +0 | データレジスタ | 文字の読み書き |
| +1 | 割り込み許可 | 割り込みの有効/無効 |
| +2 | FIFO制御 | バッファの設定 |
| +3 | ライン制御 | データフォーマット（8N1など） |
| +4 | モデム制御 | DTR, RTS信号 |
| +5 | ライン状態 | 送信可能か、データありかの確認 |

### COMポートのアドレス

| ポート | ベースアドレス |
|--------|---------------|
| COM1 | 0x3F8 |
| COM2 | 0x2F8 |
| COM3 | 0x3E8 |
| COM4 | 0x2E8 |

### 文字を送る手順

1. ライン状態レジスタ(+5)のビット5を確認 → 送信バッファが空か？
2. 空になったら、データレジスタ(+0)に1バイト書き込む
3. 繰り返し！

### 設計のポイント: PortIOトレイト

```rust
/// テスト時はモック、実機ではx86のin/out命令
pub trait PortIO {
    fn read(&self, port: u16) -> u8;
    fn write(&self, port: u16, value: u8);
}
```

**なぜトレイトを使うのか？**
- テスト時: MockPortIO（メモリ上でシミュレート）
- 実機: X86PortIO（in/out アセンブリ命令）
- 同じSerialPortコードで両方に対応できる！

### MINIX 3との比較

| 項目 | MINIX 3 | 自作OS（現在） |
|------|---------|---------------|
| ドライバの場所 | ユーザー空間 (`drivers/rs232/`) | カーネル内 |
| I/Oアクセス | カーネル経由のシステムコール | 直接 in/out 命令 |
| 障害耐性 | ドライバクラッシュ → 再起動可能 | ドライバクラッシュ → カーネル停止 |
| テスト | ハードウェア必要 | PortIOトレイトでモック可能 |

**今後の改善**: Phase 3以降でユーザー空間ドライバに分離予定

### 初期化シーケンス

```
1. 割り込み無効化（IER = 0x00）
2. DLAB有効化（LCR = 0x80）→ ボーレート設定モード
3. ボーレート 115200 設定（divisor = 1）
4. 8N1データフォーマット設定（LCR = 0x03）→ DLAB無効化
5. FIFO有効化、14バイト閾値（FCR = 0xC7）
6. モデム制御（MCR = 0x0B）→ DTR + RTS + OUT2
```

### 実装ファイル

- `src/kernel/src/serial.rs` - UARTドライバ本体（テスト10件）
- `src/kernel/src/main.rs` - X86PortIO（実機用）とカーネル統合
